    4. Authentication & Authorization
    ---------------------------------
        1. OAuth 2.0 vs OAuth 1.0 // use built in Identoty provider like Microsoft, Google etc.
            By strict definition, the OAuth 2.0 framework provides only two primary tokens: Access Tokens and Refresh Tokens.
            Here we get Access & Refresh Token. 
            Refresh token are stored in Server to cancel then token when logouts. And stored in UI so it can refresh the new access token in every interval.
            Both needs to be sent as Authorization: Bearer token 
            Access token stored in UI and server can decryot using secretKey. In Ui, stire in cookie as HttpOnly true 
        2. Token Types
            1. Access Token 
                // For Authorization, used by API server.
                When your app needs data from a server (like fetching a list of orders), it uses the Access Token.
            2. Refresh Token - part of OAuth2.0 

            3. ID Token - part of OIDC OpenID connect
                // For Authentication, used by client side, contains name, email, issuer.
                While you often receive an ID Token during this process, it actually belongs to the OpenID Connect (OIDC) layer, 
                which sits on top of OAuth 2.0 to provide user identity. 
                Your app receives the ID Token, decodes it (it’s just a JSON object), and uses the data to set up the UI.

        3. Token Formats
            1. Bearer Token: It says who is owning the token. JWT and API key can be sent as Bearer 
            2. JWT Token: it is format of token which holdd data, have 1. issuer 2. payload 3. Signature 
            3. API Keys: it is long lived string key which does nto expire until revoked. usually used to access public API or server-to-server communication.
            4. Authorization Codes: In OAuth2.0 process: After a user logs in, the server sends this code to your app. Your app then immediately exchanges this code for an Access Token and a Refresh Token.
        4. OIDC : opene id connect : OpenID Connect or OIDC is an identity protocol that utilizes the authorization and authentication mechanisms of OAuth 2.0.

    13. CQRS pattern:
        Why Read Queries are Cheaper than Writes
            A read query is less resource-intensive because it only retrieves data, whereas a write query must actively change the state of the database. 

            Indexing Overhead: Every time you write data, the database must also update all related indexes, which multiplies the work for a single command. 
                                Reads use these indexes to find data instantly without additional processing.
            Locking & Contention: Writes require exclusive locks, which block other processes from accessing that data to ensure integrity. 
                                Read operations typically use shared locks or none at all (depending on isolation levels), allowing many users to read the same data simultaneously without waiting.
            Caching & I/O: Most databases can fulfill popular reads entirely from RAM (Buffer Pool). A write must eventually reach persistent storage (SSD/Disk) to be considered durable, 
                                which is orders of magnitude slower than reading from memory.
        
        Read and write DB will not same table definition & data:
            The data in your read database is a projection of the write database, not a mirror image.
                Subset of Data: The Read DB often only stores the fields needed for the UI. For example, the Write DB might store sensitive user audit logs, 
                                while the Read DB only stores the public profile information.
                Pre-Calculated Values: If you need to show a "Total Spent" for a user, the Write DB stores individual orders. 
                                    The Read DB might store a pre-calculated TotalSpent value that is updated every time a new order event occurs, making the read instant. 

        For read you can use No-Sql or Redis database for faster execution.

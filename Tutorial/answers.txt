        8. Task & Thread 
            Task.Run(()=>{Console.WriteLine("running tasks")});
            It creates and starts the task immediately by queuing it to the default TaskScheduler (which uses the thread pool) in a single operation.
            It is the standard and safest approach for general-purpose parallel or asynchronous operations.

            Task.Start()
            Creating a Task with its constructor and then calling .Start() requires manual control over the task's lifecycle.
            var task = new Task(()=>{Console.WriteLine("running new task")});
            task.Start();
            The constructor instantiates the task in the Created state, but does not execute it until .Start() is explicitly called. This separation allows for conditional execution or managing a set of tasks before running them.

            The difference between these two methods is that Task.Run is a static method that both instantiates and runs a task, whereas Task.Start is an instance method, which you run on an existing Task instance.
            So if there's a reason you need to separate the instantiation of the task from the starting of the task (for example if you don't want to run it yet) then use new Task() with and then start it later with .Start(), otherwise just use Task.Run to do both in one call.

            4. Abstract class

                We can not create the object of abstarct class directly but we can create the instance of subclass which technically access all the properties fo abstract class.
                
                Safety: Making the class abstract prevents anyone from accidentally creating a generic, non-functional object. 
                Mandatory Implementation: If you mark a method as abstract, every subclass must provide its own version, or it won't compile. 
                    This ensures that essential functionality (like calculateSalary() for an Employee) is never forgotten.
                Through this you can achieve inheritance as well.
                Separation of Concerns: It clearly distinguishes between what an object must do (abstract methods) and how it provides common functionality (concrete methods).

    4. Asynchronous programming
    -----------------------------
        1. why to use?
            By default the c# program is synchronous but we may have some tasks which is dependent on external factor and may take little more time to be completed. 
            The problem is that, the main thread will be blocked until the task gets complete because of the synchronous behavior. 
            and due to this the application will be freezed so we need asynchronous process which will release the main thread to complete the other tasks.

        2. Multiple async handling
            put all the asynchronous task in separate Task and use Task.WhenAll(T1, t2, T3).
            
                // 1. Start all asynchronous tasks concurrently
                Task<UserData> userTask = FetchUserDataAsync();
                Task<ConfigData> configTask = LoadConfigurationAsync();
                Task<List<Product>> productsTask = GetProductListAsync();

                // 2. Await all of them to complete simultaneously
                await Task.WhenAll(userTask, configTask, productsTask);

                // 3. Process results once all tasks are finished
                stopwatch.Stop();
                Console.WriteLine($"All tasks completed in {stopwatch.ElapsedMilliseconds} ms.");

                UserData user = await userTask;
                ConfigData config = await configTask;
                List<Product> products = await productsTask
                
        3. Task.Start vs Task.Run()
            Task a = new Task(()=>{}); // when you created the task it is in Created stated and not run yet.
            a.start() // it is two step process and task will be running only when you hit a.Start()

            Task.Run() // starts the task instantly and we do need to manually handle the Task.

        4. Async Await 
            sometime when asynchronous task is there and its taking time so the main thread will be released and on next line if the code is dependent on the result of the async task then 
            it amy break the code because the result might be null until the response comes.
            so in that case, we should use await keyword alongside async to make sure next line of code will be executed only after the response comes.
            async should be in front of the method and await should be on front of the line.

            A C# application's default execution is managed by a single main thread created by the Common Language Runtime (CLR) at startup
                The .NET thread pool is an automatically managed pool of worker threads. It is not used for the initial execution of the Main() method. Instead, it is utilized when you explicitly queue work to it, or when using mechanisms like:
                    1. Asynchronous methods using async and await keywords (when a continuation is scheduled)
                    2. Task Parallel Library (TPL) methods (e.g., Task.Run())
